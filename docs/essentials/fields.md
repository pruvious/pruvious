# Fields

Fields are the building blocks of every collection, singleton, and block in Pruvious. A field is a single piece of data - a title, a price, an image, a list of tags - together with the rules that govern it: its database column type, default value, sanitizers, validators, populator, and how it renders in the dashboard.

You declare fields in TypeScript. Pruvious takes that declaration and turns it into:

- A column (or columns) in your database
- A typed entry on the query builder
- A REST API parameter
- A form input in the dashboard

```ts
// server/collections/Posts.ts
import { defineCollection, textField, richTextField } from '#pruvious/server'

export default defineCollection({
  fields: {
    title: textField({ required: true }),
    body: richTextField({}),
  },
})
```

## How a field maps to the database

Every field is backed by a **field model** that picks a primitive data type:

- `bigint` -> stored as a `BIGINT` column
- `boolean` -> stored as a `BOOLEAN` column
- `numeric` -> stored as a `NUMERIC` column
- `text` -> stored as a `TEXT` column
- `junction` and `matrix` -> stored using auxiliary tables (used by relation fields)

When you add a field to a collection at the top level, Pruvious creates the corresponding column on the next server start. When you remove a field, the column is dropped - the database tracks your code.

Nested fields (subfields of an `objectField`, items inside a `repeaterField`, or props on a block) are not stored as separate columns. They are serialized as JSON into the parent field's text column.

## Common options

Every field accepts a common set of options regardless of its type:

```ts
textField({
  required: true,
  default: 'Untitled',
  immutable: false,
  unique: true,
  sanitizers: [(value) => (typeof value === 'string' ? value.trim() : value)],
  validators: [
    (value, { context }) => {
      if (value === 'Voldemort') {
        throw new Error(context.__('pruvious-api', 'You know who'))
      }
    },
  ],
  conditionalLogic: { type: { '=': 'article' } },
  dependencies: ['./subtitle'],
  ui: {
    label: ({ __ }) => __('pruvious-dashboard', 'Title'),
    description: 'Shown in the browser tab',
    placeholder: 'My great post',
  },
})
```

| Option | Description |
| --- | --- |
| `default` | Initial value when no value is provided. Falls back to the model's default value. |
| `required` | Whether the field must be present on insert. Read-only on updates unless explicitly cleared. |
| `nullable` | Whether `null` is a valid value (most field types fix this internally). |
| `immutable` | If `true`, the value is locked after the initial insert. |
| `unique` | Enforces uniqueness via a database index. Can be a boolean or `{ caseSensitive, perLanguage, errorMessage }`. |
| `autoGenerated` | Marks the field as derived (e.g. `createdAt`). Users cannot supply a value. |
| `sanitizers` | Functions that transform the input before validation. |
| `validators` | Functions that throw or write into `errors` when input is invalid. |
| `guards` | Validator-like functions executed only by the `guarded*` query builders, used for access control. |
| `populator` | Server function that transforms the casted value into the populated value returned by the API. |
| `dependencies` | Field paths that become required when this field is provided. See [Dependencies](#dependencies). |
| `conditionalLogic` | Rules that control whether validation runs based on other field values. See [Conditional logic](#conditional-logic). |
| `inputFilters` | Hooks that run at specific points in the insert/update pipeline (`beforeInputSanitization`, `beforeInputValidation`, `beforeQueryExecution`). |
| `ui` | Dashboard appearance options. See [The `ui` block](#the-ui-block). |

Per-field-type options live alongside these (e.g. `minLength` on `textField`, `choices` on `selectField`). See the [Field reference](../fields/index.md) for what each type accepts.

## The `ui` block

`ui` controls how a field is rendered in the dashboard. It has nothing to do with what is stored in the database.

```ts
textField({
  ui: {
    label: ({ __ }) => __('pruvious-dashboard', 'Title'),
    description: 'The page title shown in browser tabs',
    placeholder: 'Untitled',
    hidden: false,
    disabled: false,
    dataTable: {
      visible: true,
      sortable: true,
      filterable: true,
    },
  },
})
```

| `ui` option | Description |
| --- | --- |
| `label` | Field label. Pass a string or a callback `({ __ }) => __('pruvious-dashboard', 'Title')` for translations. If omitted, the field name is title-cased. |
| `description` | Help text shown under the field. Supports basic markdown, plus an expandable variant with `{ text, showLabel, hideLabel, expanded }`. |
| `placeholder` | Placeholder text. Only available when the field type opts in. |
| `hidden` | Hide the field in forms (still visible in data tables and via the API). |
| `disabled` | Render as read-only in the dashboard form. Cascades into subfields. |
| `dataTable` | Configure visibility, sortability, and filterability in the data table. Pass `false` to drop the column entirely. |
| `customComponent` | Override the form input with your own Vue component. Resolve via `resolvePruviousComponent('>/components/MyField.vue')`. |
| `customTableComponent` | Override the data table cell renderer. |
| `customFilterComponent` | Override the data table filter input. |

Translatable values (`label`, `description`, `placeholder`) take a callback so the dashboard re-renders them when the user switches language:

```ts
ui: {
  label: ({ __ }) => __('pruvious-dashboard', 'First name'),
  description: ({ __ }) => __('pruvious-dashboard', 'Used for greetings in emails'),
}
```

Use a plain string when the label is not translatable.

## Sanitizers and validators

Sanitizers transform a value before validation runs. Validators decide whether the resulting value is acceptable.

The pipeline for a single field on insert or update is:

1. `inputFilters.beforeInputSanitization` (if set)
2. Model sanitizers (e.g. cast to string, trim)
3. `defineField`-level sanitizers (set when defining the field type)
4. Field-level sanitizers (set on the field instance, what you write in your collection)
5. `inputFilters.beforeInputValidation` (if set)
6. Model validators (e.g. `required`, `minLength`)
7. `guards` (only when using `guardedQueryBuilder`)
8. `defineField`-level validators
9. Field-level validators
10. Uniqueness validator (auto-injected at the end of `field.validators` when `unique: true`)
11. `inputFilters.beforeQueryExecution` (if set)
12. Database write

A validator can either throw an `Error` whose message becomes the field error, or write into the shared `errors` object using dot-notation keys (useful for surfacing multiple errors at once).

```ts
import { textField } from '#pruvious/server'

textField({
  sanitizers: [(value) => (typeof value === 'string' ? value.toLowerCase() : value)],
  validators: [
    (value, { context }, errors) => {
      if (typeof value === 'string' && value.includes(' ')) {
        errors['name'] = context.__('pruvious-api', 'Spaces are not allowed')
      }
    },
  ],
})
```

## Conditional logic

`conditionalLogic` lets a field opt in or out of `required` validation based on other field values. The keys are paths relative to the field; the values are operator/value pairs.

```ts
textField({
  required: true,
  conditionalLogic: { type: { '=': 'article' } },
})
```

Path syntax:

- `fieldName` or `./fieldName` - sibling field
- `address.city` - nested field
- `..` - parent (for arrays, this is the array length)
- `../color` - sibling of parent
- `/layout` - root-level field

Operators: `=`, `!=`, `>`, `>=`, `<`, `<=`, `regexp`.

Combine multiple conditions:

```ts
// AND - all conditions must match
conditionalLogic: {
  size: { '=': 'sm' },
  '../color': { regexp: '^(?:[a-z]+-)?blue$' },
}

// OR - any group must match
conditionalLogic: {
  orGroup: [
    { size: { '=': 'sm' } },
    { '../color': { '=': 'red' } },
  ],
}
```

When the condition does not match, the field's `required` rule is skipped. Other validators still run if a value is supplied.

## Dependencies

`dependencies` is the opposite direction: it makes **other** fields required when **this** field has a value.

```ts
imageField({
  dependencies: ['./alt'],
})
```

If `image` is set but `alt` is empty, the request fails. Paths follow the same syntax as `conditionalLogic`.

## Field layout

By default the dashboard stacks fields vertically in the order they appear in your collection. Use the `fieldsLayout` option on the collection (and `subfieldsLayout` on an `objectField` or `repeaterField`) to group them into rows, cards, and tabs.

```ts
import { defineCollection, textField, selectField } from '#pruvious/server'

export default defineCollection({
  fields: {
    firstName: textField({}),
    lastName: textField({}),
    countryCode: textField({}),
    phone: textField({}),
    notes: textField({}),
    role: selectField({ choices: [/* ... */] }),
  },
  fieldsLayout: [
    // Row of two equal fields
    { row: ['firstName', 'lastName'] },

    // Fixed-width field plus a flexible one
    { row: ['countryCode | 8rem', 'phone'] },

    // Horizontal separator
    '---',

    // Tabbed groups
    {
      tabs: [
        {
          label: ({ __ }) => __('pruvious-dashboard', 'Profile'),
          fields: ['role'],
        },
        {
          label: ({ __ }) => __('pruvious-dashboard', 'Internal'),
          fields: ['notes'],
        },
      ],
    },

    // Collapsible card
    {
      card: {
        collapsible: true,
        header: ({ __ }) => __('pruvious-dashboard', 'Advanced'),
        fields: ['notes'],
      },
    },
  ],
})
```

Layout primitives:

- `'fieldName'` - a single field, full width
- `'fieldName | 50%'` or `'fieldName | 8rem'` - cap the field's maximum width
- `'fieldName | auto'` - shrink to content
- `{ field: { name, style } }` - inline Vue styles
- `{ row: [...] }` - horizontal group
- `{ card: [...] }` or `{ card: { collapsible, header, fields } }` - card group
- `{ tabs: [{ label, queryParam?, fields }, ...] }` - tabbed groups
- `'---'` or `{ hr: true }` - horizontal rule
- `{ component: resolvePruviousComponent('>/components/MyPanel.vue') }` - drop in a Vue component

You can name fields anywhere in `row`, `card`, or `tabs` - TypeScript validates them against the collection's field keys.

## Defining custom fields

To create a new field type, add a file under `server/fields/`. The filename (camelCase) becomes the field name.

```ts
// server/fields/color.ts
import { defineField } from '#pruvious/server'
import { textFieldModel } from '@pruvious/orm'

const customOptions: {
  /**
   * Allowed color formats.
   *
   * @default ['hex', 'rgb']
   */
  formats?: ('hex' | 'rgb' | 'hsl')[]

  ui?: {
    /**
     * Show a color preview swatch next to the input.
     *
     * @default true
     */
    showSwatch?: boolean
  }
} = {
  formats: ['hex', 'rgb'],
  ui: { showSwatch: true },
}

export default defineField({
  model: textFieldModel(),
  customOptions,
  uiOptions: { placeholder: true },
  validators: [
    (value, { definition, context }) => {
      if (typeof value !== 'string') {
        return
      }
      const { formats } = definition.options
      const hex = /^#[0-9a-f]{3,8}$/i
      if (formats?.includes('hex') && hex.test(value)) {
        return
      }
      throw new Error(context.__('pruvious-api', 'Invalid color format'))
    },
  ],
})
```

Once saved, the field is auto-imported and available as `colorField` from both `#pruvious/server` and `#pruvious/app`:

```ts
import { colorField } from '#pruvious/server'

colorField({
  formats: ['hex'],
  ui: { showSwatch: false, label: 'Brand color' },
})
```

Field components (`app/fields/color.vue`, `app/fields/color.table.vue`, `app/fields/color.filter.vue`) are picked up automatically when present.

Reserved option names you cannot use in `customOptions`: `autoGenerated`, `conditionalLogic`, `default`, `dependencies`, `immutable`, `nullable`, `required`, `unique`. Anything starting with `_` is reserved for internal use.

## Field presets

Pruvious ships a handful of helper fields for patterns that appear in nearly every collection. They wrap an underlying field with sensible defaults, validators, and dashboard config.

```ts
import {
  defineCollection,
  textField,
  createdAtFieldPreset,
  updatedAtFieldPreset,
  authorFieldPreset,
  isPublicFieldPreset,
  scheduledAtFieldPreset,
  seoFieldPreset,
  subpathFieldPreset,
} from '#pruvious/server'

export default defineCollection({
  fields: {
    title: textField({ required: true }),
    subpath: subpathFieldPreset({}),
    isPublic: isPublicFieldPreset({}),
    scheduledAt: scheduledAtFieldPreset({}),
    seo: seoFieldPreset({}),
    author: authorFieldPreset({}),
    createdAt: createdAtFieldPreset({}),
    updatedAt: updatedAtFieldPreset({}),
  },
})
```

| Preset | Wraps | Purpose |
| --- | --- | --- |
| `createdAtFieldPreset` | `timestampField` | Auto-set on insert. Immutable. |
| `updatedAtFieldPreset` | `timestampField` | Auto-set on every update. |
| `authorFieldPreset` | `recordField` | Links to the `Users` collection; auto-populated with the current user on insert. |
| `editorsFieldPreset` | `recordsField` | Links to `Users` for multi-editor records. |
| `languageFieldPreset` | `languageField` | Stores a record's language. Required for translatable collections. |
| `translationsFieldPreset` | `translationsField` | Unique identifier connecting translations across languages. |
| `subpathFieldPreset` | `subpathField` | URL slug for routable collections. |
| `isPublicFieldPreset` | `trueFalseField` | Draft/Public switch with labels and a sensible default. |
| `scheduledAtFieldPreset` | `dateTimeField` | Optional publish-at timestamp. Pairs with `isPublic`. |
| `seoFieldPreset` | `objectField` | Full SEO object (title, description, indexability, sharing image, canonical URL, meta tags). |

Presets are plain function calls - you can still pass options to override the defaults. Each preset accepts a `ui` block to customize its dashboard label, description, and layout.
